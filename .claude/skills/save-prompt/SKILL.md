# save-prompt スキル

直近のセッションでのユーザーとのやり取りや、実施した作業の概要を Markdown ドキュメントとして
`doc/prompt/` フォルダに保存し、`main` ブランチへ commit & push する。

## 手順

### 1. 既存ファイルを確認して連番・命名を決める

```bash
ls -1 doc/prompt/
```

- ファイル名は **`YYMMDD-連番-題名.md`** 形式。
  - `YYMMDD`: 今日の日付（例: 2026-06-10 → `260610`）。
  - `連番`: 同じ日付内で既存の最大番号 + 1（ゼロ埋め2桁、例: `01`, `02`）。
  - `題名`: 内容が一目で分かる日本語の短い題名。スペースは使わず簡潔に。
- 例: `260610-02-お問い合わせページ削除とmain直push運用.md`

### 2. ドキュメントを作成する

直近のセッションの内容をまとめて Markdown ファイルを作成する。既存ファイルの体裁に合わせる。

推奨構成:

```markdown
# 題名

## 概要

（このセッションで何をしたかを2〜4行で要約）

## 実施内容

### 1. ◯◯
（具体的な変更内容。編集・削除したファイルパスを箇条書きで明記）

### 2. ◯◯
...

## 関連コミット

- （main に入れたコミットの要約）

## 今後の運用メモ（必要な場合のみ）

- （継続する方針や注意点）
```

ポイント:
- 編集・削除・追加したファイルは具体的なパスで記載する。
- ユーザーから受けた明示的な指示・方針（例: 「今後は main に直 push」）は残す。
- 既存の `doc/prompt/*.md` のトーンと粒度に合わせる。

### 3. main ブランチへ commit & push

```bash
git add -A
git commit -q -m "作業概要ドキュメント追加: <題名>"
git push origin main
```

- このリポジトリは **main ブランチへ直接 push する運用**（ユーザー許可済み）。
- push が network error で失敗した場合のみ、指数バックオフ（2s/4s/8s/16s）で最大4回リトライ。
- pull request は作成しない。

## 注意点

- ビルド成果物（`*.tsbuildinfo` など）はコミットしない（`.gitignore` 済み）。
- 未コミット・未push の他の変更がある場合は、内容を確認してから含めるか分けるか判断する。
